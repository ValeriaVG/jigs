jigs

crates.io docs.rs github

Explicit request-to-response pipelines for Rust.

Heads up. This is an exploratory project. APIs will change, things will break, ideas will get thrown out. Play with it, don't ship it.

HTTP handlers, RPC services, agent workflows, data pipelines, event listeners: they all have the same shape. Something comes in, a handful of steps run, something goes out. The more steps there are, the harder the code gets to follow.

A jig is the tool a machinist clamps a workpiece into so the cutting tool always lands in the same place. jigs is the same idea for code: each step is named, typed, and held in a known position relative to the others.

jigs lets you spell those steps out:

#[jig]
fn handle(request: Request) -> Response {
    request
        .then(log_incoming)
        .then(set_auth_state)
        .then(route_to_feature)
        .then(log_response)
}

Each step is itself a jig, so pipelines compose. Every #[jig] registers itself in a global inventory at link time, which means the whole graph is known at compile time and you can render it without scanning sources or running a build script.

Install

cargo add jigs

For per-step timing, turn on the trace feature:

cargo add jigs --features trace

Quickstart

use jigs::{jig, Request, Response};

#[jig]
fn validate(r: Request<u32>) -> Request<u32> { r }

#[jig]
fn handle(r: Request<u32>) -> Response<String> {
    Response::ok(format!("got {}", r.0))
}

fn main() {
    let response = Request(42u32).then(validate).then(handle);
    assert_eq!(response.inner.unwrap(), "got 42");
}

There are four shapes of jig:

InputOutputUse
RequestRequestenrich, validate, transform
RequestResponseterminal handler
ResponseResponsepost-process the outgoing message
RequestBranch<Req,Resp>guard that may short-circuit

The type system enforces ordering: once you hold a Response, you can't chain a jig that expects a Request. Errored responses and Branch::Done short-circuit the rest.

For multi-arm dispatch use fork!. First matching predicate wins, _ is the default:

#[jig]
fn route(req: Request<HttpRequest>) -> Response<String> {
    fork!(req,
        |r: &HttpRequest| r.path == "/"                  => root,
        |r: &HttpRequest| r.path.starts_with("/hello/")  => hello,
        _ => not_found,
    )
}

Generate a map

Render an HTML map or a Mermaid-flavored Markdown one straight from your code:

let dir = env!("CARGO_MANIFEST_DIR");
std::fs::write(
    format!("{dir}/map.html"),
    jigs::map::to_html(Some("handle"), "my service", None),
)?;
std::fs::write(
    format!("{dir}/map.md"),
    jigs::map::to_markdown(Some("handle"), "my service"),
)?;

The third argument to to_html is an editor URL template. Some("vscode://file/{path}:{line}") for VS Code or Cursor, idea://open?file={path}&line={line} for JetBrains, or a GitHub blob URL if you want maps that link back to the repo.

Trace at runtime

With the trace feature on, every step records its timing. Drain the per-thread buffer and render it in either format:

let entries = jigs::trace::take();
print!("{}", jigs::log::render_tree(&entries));   // human-readable
print!("{}", jigs::log::render_ndjson(&entries)); // one JSON object per line

A rendered tree looks like this:

log_incoming     ok  0.1ms
set_auth_state   ok  2.3ms
route_to_feature err 500
  validate_route    ok  0.4ms
  check_permissions err DB Timeout

NDJSON is meant for log ingestion; each line carries name, depth, duration_ns, ok, and an optional error.
