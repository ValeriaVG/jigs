A Rust framework

Explicit pipelines, with interactive maps.

jigs turns your request → response code into a graph the compiler verifies and you can navigate. Every step is a typed, composable jig — and every pipeline ships with a live, clickable map.

A jig is a custom-made tool that controls the location or motion of other tools. HTTP handlers, RPC services, agentic workflows, data pipelines and event listeners all share one shape: something comes in and, after several steps, something comes out. The more steps there are, the harder it gets to debug, fix, and maintain.

jigs lets you be explicit about your steps and write code like this:

#[jig]
fn handle(request: Request) -> Response {
    request
        .then(log_incoming)
        .then(set_auth_state)
        .then(route_to_feature)
        .then(log_response)
}

Every step is a jig too, so pipelines compose to any size. jigs generates a graph of your codebase at compile time and lets you navigate it.

Install

cargo add jigs

To enable per-step tracing, turn on the trace feature:

cargo add jigs --features trace

Quickstart

use jigs::{jig, Request, Response};

#[jig]
fn validate(r: Request<u32>) -> Request<u32> { r }

#[jig]
fn handle(r: Request<u32>) -> Response<String> {
    Response::ok(format!("got {}", r.0))
}

fn main() {
    let response = Request(42u32).then(validate).then(handle);
    assert_eq!(response.inner.unwrap(), "got 42");
}

There are four kinds of jigs, distinguished by their input and output types:

InputOutputPurpose
RequestRequestenrich, validate, transform
RequestResponseterminal handler
ResponseResponsepost-process the outgoing message
RequestBranch<Req,Resp>guard that may short-circuit

The type system enforces ordering: once you hold a Response, you cannot chain a jig that expects a Request. Errored responses and Branch::Done short-circuit the rest of the pipeline.

Generate a map

Every #[jig] registers itself in a global inventory at link time, so you can render an interactive HTML map and a Markdown/Mermaid version directly from your code — no scanning, no build script.

let dir = env!("CARGO_MANIFEST_DIR");
std::fs::write(
    format!("{dir}/map.html"),
    jigs::map::to_html(Some("handle"), "my service", None),
)?;
std::fs::write(
    format!("{dir}/map.md"),
    jigs::map::to_markdown(Some("handle"), "my service"),
)?;

The third argument to to_html is an optional editor URL template. Pass Some("vscode://file/{path}:{line}") for VS Code or Cursor, idea://open?file={path}&line={line} for JetBrains IDEs, or a GitHub blob URL to publish maps that link back to your repo.

Trace at runtime

Things break, and the bigger the system, the harder it is to find where. jigs lets you pinpoint an error or a bottleneck:

log_incoming     ✓  0.1ms
set_auth_state   ✓  2.3ms
route_to_feature ✗  ERROR 500
  └─ validate_route  ✓  0.4ms
  └─ check_permissions ✗  ERROR: DB Timeout

With the trace feature on, drain the per-thread buffer in either format:

let entries = jigs::trace::take();
print!("{}", jigs::log::render_tree(&entries));   // human-readable
print!("{}", jigs::log::render_ndjson(&entries)); // one JSON object per line

The NDJSON form is meant for automated log ingestion; each line carries name, depth, duration_ns, ok and an optional error message.
